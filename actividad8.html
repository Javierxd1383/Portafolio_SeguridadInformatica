01. OBJETIVOS DE LA ACTIVIDAD

Objetivo general

Desarrollar un portafolio técnico profesional que documente la explotación sistemática de vulnerabilidades de SQL Injection en los laboratorios oficiales de PortSwigger, aplicando Burp Suite como herramienta principal, demostrando comprensión técnica, metodología estructurada y capacidad de análisis de impacto en seguridad web.

Objetivos específicos

  • Comprender y fundamentar teóricamente el funcionamiento de las vulnerabilidades de SQL Injection, sus variantes (UNION, Error-Based, Blind, Time-Based, OOB, etc.) y su impacto en la seguridad de aplicaciones web.
  • Aplicar técnicas prácticas de explotación utilizando Burp Suite para resolver los laboratorios del tópico SQL Injection en PortSwigger, documentando paso a paso cada procedimiento técnico.
  • Analizar el riesgo y las medidas de mitigación, vinculando cada vulnerabilidad con buenas prácticas de desarrollo seguro (OWASP, validación de entradas, prepared statements, etc.).

02. ESTRUCTURA DEL DOCUMENTO

Deberás elaborar un documento técnico-profesional en el que se documente de manera estructurada la resolución del tópico SQL Injection correspondiente a los laboratorios oficiales de PortSwigger (Web Security Academy).

Estructura del documento:

  1. Portada.
  2. Introducción.
  3. Marco teórico.
    • Funcionamiento de bases de datos relacionales
    • Consultas SQL básicas (SELECT, WHERE, UNION, etc.)
    • ¿Qué es SQL Injection?
    • Clasificación de ataques SQLi:
      • In-band (Union-based, Error-based)
      • Blind (Boolean-based, Time-based)
      • Out-of-band
    • Impacto en la confidencialidad, integridad y disponibilidad
    • Relación con OWASP Top 10
    • Importancia en pruebas de penetración
    • Tipos de payloads
    • Principios de defensa
  4. Desarrollo práctico.
    • LABS APPRENTICE
      • SQL injection vulnerability in WHERE clause allowing retrieval of hidden data
      • SQL injection vulnerability allowing login bypass
    • LABS PRACTITIONER
      • Querying database type/version (Oracle, MySQL, Microsoft)
      • Listing database contents (Oracle and non-Oracle)
      • UNION attacks (column count, text column, retrieving data)
      • Blind SQL injection (conditional responses, errors, time delays)
      • Out-of-band interaction
      • XML encoding filter bypass
  5. Conclusiones. (Integra captura de pantalla de tu avance / score)
  6. Referencias.

Para cada laboratorio deberá incluir:

  1. Nombre del laboratorio.
  2. Nivel (Apprentice / Practitioner).
  3. Tipo de SQLi.
  4. Objetivo del laboratorio.
  5. Explicación técnica de la vulnerabilidad.
  6. Procedimiento paso a paso.
  7. Capturas de pantallas obligatorias: request interceptado, payload utilizado, respuesta vulnerable, confirmación de "Lab Solved".
  8. Explicación del payload.
  9. Mitigación recomendada.

03. RÚBRICA DE EVALUACIÓN

Explicación técnica de los payloads 20 pts. 18-20 pts. — Explica por qué funciona cada payload, cómo modifica la lógica SQL y qué condición altera (WHERE, UNION, delays, etc.). 14-17 pts. — Explicación correcta pero limitada en profundidad. 10-13 pts. — Descripción básica sin análisis de la lógica SQL. 0-9 pts. — No explica el funcionamiento técnico del payload.

Evidencias y capturas de pantalla 15 pts. 14-15 pts. — Capturas claras, completas y organizadas (request, respuesta, "Lab Solved"); evidencias verificables. 11-13 pts. — Evidencias suficientes con pequeños errores de claridad. 8-10 pts. — Evidencias incompletas o poco legibles. 0-7 pts. — Ausencia o deficiencia grave de evidencias.
